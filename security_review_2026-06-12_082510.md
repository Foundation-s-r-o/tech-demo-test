# Security Review - Tech Demo Test

**Review date:** 2026-06-12 08:25 CEST  
**Repository:** `/Users/antony/fndt/demo/tech-demo-test`  
**Branch / commit:** `main` / `810c24c`  
**Reviewed state:** current working tree; the pre-existing `CLAUDE.md` modification was not changed  
**Overall result:** **ISSUES FOUND - not suitable for production deployment in its current configuration**

## Executive Summary

The application has a useful authentication baseline: protected business endpoints return 401 anonymously, passwords use BCrypt, login errors do not enumerate accounts, JPA queries avoid string-built SQL, React does not use obvious unsafe HTML sinks, and the existing unit, integration, Playwright, smoke, lint, and build checks pass.

The production security boundary is nevertheless unsafe. Runtime testing confirmed that an anonymous caller can inspect all Actuator endpoints and change the application's root logging level. The API accepts credentialed requests from arbitrary origins while CSRF is disabled. A cross-origin authenticated POST successfully created data without a CSRF token. The manually implemented login does not rotate the existing session ID, which leaves session fixation protection incomplete. The dependency graph also contains a current critical command-injection advisory in developer/build tooling.

Deployment files amplify the risk: they pass an administrator password as a Docker build argument, use long-lived AWS keys and old unpinned GitHub Actions, run the final image as root, and include a privileged infrastructure container with host PID, host networking, the host root filesystem, and the Docker socket. CI does not run security scans or frontend linting and does not block the known critical dependency.

## Detailed Findings

### SEC-01 - Critical command-injection advisory in build tooling

**Severity:** Critical  
**Status:** ISSUES FOUND  
**Evidence:** `ui/package-lock.json`; `npm audit`; Trivy `CVE-2026-9277`; GitHub advisory `GHSA-w7jw-789q-3m8p`

`shell-quote@1.8.3` is present through `concurrently@9.2.1`, which is used by the direct `@openapitools/openapi-generator-cli` dependency. Version `1.8.4` contains the fix. The vulnerable path is developer/build tooling rather than the Java runtime, but command injection in CI or a developer workstation still has access to source, build credentials, and CI tokens. `npm audit` reports three critical affected package entries arising from this underlying advisory; Trivy reports one critical CVE.

**Remediation:** update the direct tooling dependencies or add a temporary lockfile override to force `shell-quote>=1.8.4`; then regenerate the lockfile and run the complete frontend suite. Do not waive this solely because it is a development dependency.

### SEC-02 - Anonymous Actuator inspection and runtime modification

**Severity:** High  
**Status:** ISSUES FOUND - dynamically confirmed  
**Evidence:** `api/src/main/resources/application.properties:51`; `api/src/main/java/sk/foundation/techdemo/auth/SecurityConfig.java`

All Actuator endpoints are exposed and the entire `/actuator/**` path is public. Anonymous requests returned environment keys, configuration properties, mappings, beans, metrics, thread dumps, SBOM data, and logger controls. Values such as the datasource password were masked, which is positive, but endpoint names, application structure, and operational metadata remain exposed.

An anonymous `POST /actuator/loggers/ROOT` changed the root logger from `INFO` to `DEBUG` and returned HTTP 204. The review restored it to `INFO` and verified the restoration. This allows unauthenticated operational modification, log-volume denial of service, and increased sensitive diagnostic output.

**Remediation:** expose only `health` and optionally minimal `info` publicly. Put every other endpoint on a separate management interface or require an operations role and network-level restriction. Disable write operations unless explicitly required.

### SEC-03 - Arbitrary credentialed CORS combined with disabled CSRF

**Severity:** High  
**Status:** ISSUES FOUND - dynamically confirmed  
**Evidence:** `SecurityConfig.java` CORS configuration and `.csrf(csrf -> csrf.disable())`

The server reflects an arbitrary origin, allows credentials, permits all methods and headers, and disables CSRF. A login from `https://evil.example` received credentialed CORS approval. With the authenticated cookie, a cross-origin POST to `/api/persons` succeeded with HTTP 201 without a CSRF token. Cross-origin logout also succeeded.

Modern browser default `SameSite=Lax` behavior may limit some cross-site cookie transmission, but the application does not set an explicit SameSite policy. Same-site sibling origins, browser variations, embedded clients, proxy changes, or a future cookie configuration can make the exploit fully reachable. CORS is not a substitute for CSRF protection.

**Remediation:** use an explicit environment-specific allowlist of exact HTTPS origins, enable Spring Security CSRF protection, and use a supported token strategy such as `CookieCsrfTokenRepository` for the SPA. Add negative tests for untrusted origins and missing/invalid tokens.

### SEC-04 - Session ID is not rotated after authentication

**Severity:** High  
**Status:** ISSUES FOUND - dynamically confirmed  
**Evidence:** `api/src/main/java/sk/foundation/techdemo/auth/AuthController.java:36-49`

The login controller manually authenticates, creates a `SecurityContext`, and saves it to the session repository. It does not invoke Spring Security's session authentication strategy. A session created before login retained exactly the same `JSESSIONID` after successful authentication.

**Remediation:** use Spring Security's normal authentication filter flow or explicitly invoke a `SessionAuthenticationStrategy` that calls `changeSessionId`. Add an integration test asserting that the ID changes and the old ID cannot access authenticated endpoints.

### SEC-05 - Known bootstrap administrator credentials can become a deployment credential

**Severity:** High  
**Status:** ISSUES FOUND  
**Evidence:** `api/src/main/resources/db/migration/V3__create_app_user.sql`; local test login `admin/admin`; Docker and workflow credential arguments

The migration seeds a known administrator account. The local application accepted `admin/admin`. There is no explicit production profile that prevents demo credentials, H2 console configuration, public diagnostics, or insecure defaults from reaching a shared environment.

**Remediation:** remove the fixed account from normal migrations. Provision the initial administrator through a one-time deployment process, require a strong secret from a runtime secret manager, and fail production startup when demo settings or missing required secrets are detected.

### SEC-06 - Deployment secrets and GitHub Actions trust are weak

**Severity:** High  
**Status:** ISSUES FOUND  
**Evidence:** `.github/workflows/push-to-ecr.yml:27-50`; deployment workflow; `Dockerfile:15-22`

The ECR workflow uses long-lived AWS access keys and passes the administrator password via `--build-arg`, after which the Dockerfile promotes it to an environment variable during the UI build. Build arguments and build environment data can leak through image metadata, cache, provenance, or logs. Checkout and AWS actions use old moving tags rather than immutable commit SHAs. A manually supplied Git ref controls what code is built.

**Remediation:** use GitHub OIDC with narrowly scoped AWS roles and protected deployment environments. Never provide runtime credentials during image build; inject them only at runtime from a secret manager. Pin every third-party action to a full commit SHA and constrain deployable refs/tags.

### SEC-07 - Privileged observability container provides host-level access

**Severity:** High  
**Status:** ISSUES FOUND  
**Evidence:** `docker-compose.yml:119-132`

The New Relic infrastructure service is privileged, joins host PID and network namespaces, mounts `/` and the Docker socket, and uses a floating `latest` image. Compromise of this container is effectively compromise of the Docker host.

**Remediation:** remove the stale service if unused. Otherwise pin a digest, apply the vendor's least-privilege deployment, remove the Docker socket where possible, and isolate observability deployment from the application compose file.

### SEC-08 - Container executes as root and deployment definition is stale

**Severity:** High  
**Status:** ISSUES FOUND  
**Evidence:** `Dockerfile`; Trivy `DS-0002`

The final image defines no `USER`, so the Java process runs as root. It also lacks a health check, uses Java 17 while the Maven project and CI use Java 21, uses Node 18 while the UI requires Node 22 or newer, and contains invalid `ARG PLAYWRIGHT_SLOW_MO 1000` syntax that caused partial Dockerfile parsing in Semgrep. This reduces confidence that the reviewed deployment path is both buildable and representative.

**Remediation:** align the image toolchain with the supported versions, use `ARG NAME=default`, create a dedicated non-root runtime user, set a read-only filesystem where practical, drop capabilities, and add a health check. Build and scan the resulting image in CI.

### SEC-09 - CI does not enforce the documented or necessary security gates

**Severity:** High  
**Status:** ISSUES FOUND  
**Evidence:** `.circleci/config.yml:14-23,34-53`; `SECURITY.md`; `README.md`

CI runs Maven verification and the frontend build, but not frontend linting, Playwright tests, `npm audit`, Trivy, Semgrep, a container scan, or a dependency policy. It installs frontend packages using `npm install --legacy-peer-deps --no-audit`, not reproducible `npm ci`. The current critical advisory can therefore pass CI. Documentation incorrectly states that lint and type checks are CI hard gates and that critical/high Trivy findings are zero.

**Remediation:** make dependency, secret, SAST, lint, test, and image scans blocking at an agreed threshold. Fix the lockfile so `npm ci` works on Linux rather than weakening installation semantics.

### SEC-10 - No role-based authorization for business operations

**Severity:** Medium  
**Status:** ISSUES FOUND  
**Evidence:** `SecurityConfig.java`; person controller integration tests

All non-public endpoints require authentication, but no URL or method authorization distinguishes `ADMIN` and `USER`. Existing tests demonstrate that a normal mock user can perform CRUD. This is a latent privilege issue if non-administrator users are provisioned. The exact business impact depends on the intended role matrix, which is not documented.

**Remediation:** define the authorization matrix first, then enforce it with `@EnableMethodSecurity` and `@PreAuthorize` or explicit request rules. Treat frontend route restrictions only as UX, never as authorization.

### SEC-11 - No login throttling or lockout control

**Severity:** Medium  
**Status:** ISSUES FOUND - dynamically confirmed  

Twenty consecutive invalid login attempts all returned 401; none was delayed or rejected with 429. Combined with a known default administrator username, this enables online password guessing.

**Remediation:** add rate limits per account and source, progressive delay, monitoring, and an operational response. Avoid permanent account lockout that can itself be abused for denial of service.

### SEC-12 - Session cookie and response-header hardening is incomplete

**Severity:** Medium  
**Status:** ISSUES FOUND  

The observed `JSESSIONID` had `HttpOnly` and `Path=/`, but no explicit `Secure` or `SameSite` attribute. Responses included `X-Content-Type-Options` and `X-Frame-Options`, but no Content Security Policy, Referrer Policy, or Permissions Policy. HSTS cannot be meaningfully validated over the local HTTP test.

**Remediation:** set `Secure`, `HttpOnly`, and an explicit SameSite policy in the production profile; use HTTPS exclusively; enable HSTS behind a correctly configured trusted proxy; add a CSP compatible with the corrected frontend build.

### SEC-13 - Production frontend embeds source and uses eval

**Severity:** Medium  
**Status:** ISSUES FOUND  
**Evidence:** `ui/webpack.config.js:31`; generated `ui/dist/bundle.js`

`devtool: 'eval-source-map'` is unconditional. The production build contains `eval(...)`, inline source maps, and original source text. This discloses implementation details and prevents a strong CSP without allowing `unsafe-eval`. No administrator password or similarly named secret was found in the generated bundle.

**Remediation:** disable source maps in production or generate external hidden source maps uploaded only to the error-monitoring service. Ensure production code contains no `eval` and enforce CSP in tests.

### SEC-14 - Security coverage reporting is misleading and unenforced

**Severity:** Medium  
**Status:** ISSUES FOUND  

The JaCoCo report shows approximately 21.6% line coverage and zero coverage for key security classes. The report goal runs before Failsafe integration tests, so the generated report omits relevant integration execution. There is no minimum threshold. Current auth tests cover successful login, wrong password, and anonymous access only.

**Remediation:** generate the aggregate report after integration tests, set an achievable threshold, and add targeted security tests listed below. Measure behavior and branch coverage around the security boundary rather than pursuing a vanity percentage.

### SEC-15 - Security documentation contains materially stale claims

**Severity:** Medium  
**Status:** ISSUES FOUND  

`SECURITY.md`, `README.md`, `PROJECT_OVERVIEW.md`, and `codebase_analysis.md` disagree with current code and runtime behavior. Examples include zero high/critical dependency claims, CI lint-gate claims, H2 console exposure, configuration filename, application port, and whether backend authentication exists.

**Remediation:** update documentation after the security changes and add a dated evidence section generated from CI. Avoid static statements such as “zero vulnerabilities” without a scan date and artifact link.

### SEC-16 - Public API documentation expands reconnaissance

**Severity:** Medium  
**Status:** ISSUES FOUND  

Swagger UI and `/v3/api-docs` are public. This is acceptable for an intentionally public API, but the repository does not document that decision. For an internal CRUD application, it unnecessarily exposes the complete request surface.

**Remediation:** make the decision explicit. Disable API documentation in production or require authenticated developer/operations access.

### SEC-17 - Tracked browser storage-state file could leak future sessions

**Severity:** Low  
**Status:** ISSUES FOUND  
**Evidence:** `ui/storageStateFresh.json`

The tracked file is currently empty, so no credential was exposed. Its location is not covered by the existing storage-state ignore pattern. A future Playwright run could persist cookies or local storage into a tracked file.

**Remediation:** remove the tracked file and ignore all generated Playwright storage-state artifacts regardless of directory.

### SEC-18 - Email validation is syntactic only by size

**Severity:** Low  
**Status:** ISSUES FOUND  

The backend enforces non-null and maximum length but does not apply email-format validation. This is primarily a data-quality issue unless downstream systems later place the value into mail headers or templates.

**Remediation:** add `@Email` if the field is semantically an email address and keep downstream output encoding context-specific.

### SEC-19 - H2 console configuration and documentation do not match runtime

**Severity:** Low  
**Status:** PARTIAL

The H2 console is enabled and its path is listed as public in security configuration/documentation, but the tested `/h2-console` request returned 401. This reduces immediate exposure but indicates a brittle, untested configuration. H2 should not be enabled in production regardless.

**Remediation:** put H2 console enablement and frame relaxation in a local-only profile, remove its production permit rule, and add profile-specific startup tests.

## Prioritized Remediation Plan

Each batch should be a separate change set. Run its listed tests before moving to the next batch. Do not combine all security changes into a single large patch.

### Batch 1 - Block the known critical dependency

- Upgrade `shell-quote` transitively by updating `concurrently` / OpenAPI Generator CLI or use a temporary audited override.
- Regenerate and review `package-lock.json`.
- Add blocking `npm audit --audit-level=high` or an equivalent policy with explicit, expiring exceptions.
- Verify: `npm ci`, `npm audit`, `npm run lint`, `npm run build`, auth Playwright, both smoke suites.

### Batch 2 - Close the HTTP management and browser boundary

- Restrict public Actuator to health-only; secure or isolate all other endpoints.
- Replace wildcard-pattern CORS with exact per-environment origins.
- Enable CSRF using a SPA-compatible token repository.
- Add explicit cookie attributes and security headers.
- Disable Swagger and H2 outside local development.
- Verify with MockMvc/integration tests and a malicious-origin Playwright fixture.

### Batch 3 - Correct session and authentication lifecycle

- Move login into the standard Spring Security authentication flow or invoke the session strategy explicitly.
- Rotate session IDs and limit concurrent sessions if required.
- Add rate limiting and security-event logging.
- Remove fixed production bootstrap credentials and add production startup guards.
- Verify ID rotation, old-cookie rejection, logout, generic failures, and throttling.

### Batch 4 - Define and enforce authorization

- Document which roles may read, create, update, delete, export PDF, and use operations endpoints.
- Enable method security and enforce the matrix server-side.
- Add negative tests for every role/action combination.
- Keep frontend visibility rules aligned, but do not treat them as enforcement.

### Batch 5 - Harden the frontend artifact

- Make webpack configuration mode-aware and remove production `eval-source-map`.
- Deploy a CSP that does not require `unsafe-eval`.
- Clear stale local auth state on 401 and centralize expired-session handling.
- Remove and ignore generated Playwright storage state.
- Verify built assets contain no `eval(`, inline source maps, credentials, or private source.

### Batch 6 - Replace unsafe deployment scaffolding

- Align Java 21 and Node 22 across local, CI, and Docker builds.
- Run as a non-root user, add a health check, drop capabilities, and use a minimal pinned base image.
- Remove credentials from build arguments and use runtime secret injection.
- Replace AWS keys with GitHub OIDC; protect environments and constrain refs.
- Pin third-party actions and container images by immutable SHA/digest.
- Remove or isolate the privileged infrastructure agent.
- Build the image in CI and scan the final image, not only source files.

### Batch 7 - Make security evidence reliable

- Fix JaCoCo lifecycle ordering so integration tests appear in coverage.
- Add blocking security tests and scans to CI.
- Update `SECURITY.md`, README, project overview, and codebase analysis from actual configuration.
- Publish dated scan artifacts/SBOMs and document exception ownership and expiry.

## Recommended Security Tests

### Backend integration tests

1. Assert every business endpoint returns 401 anonymously.
2. Assert the agreed USER/ADMIN matrix for GET, POST, PUT, DELETE, PDF, and operations endpoints.
3. Assert state-changing requests without a CSRF token return 403 and valid tokens succeed.
4. Assert untrusted origins receive no CORS approval; assert only exact configured origins work with credentials.
5. Assert the session ID changes after login and the pre-login ID cannot authenticate.
6. Assert logout invalidates the session and repeated logout is harmless.
7. Assert only `/actuator/health` is public and logger/config/environment endpoints return 401/403.
8. Assert production cookies contain `Secure`, `HttpOnly`, and the selected SameSite value.
9. Assert wrong username and wrong password remain indistinguishable in status, body, and approximate timing.
10. Assert rate limits return 429 after the defined threshold and emit a security event.
11. Assert the production profile refuses H2 console/demo credentials and fails fast on missing required secrets.
12. Fuzz malformed JSON, oversized fields, invalid paging/sorting, and enum values; assert bounded generic 4xx responses.

### Frontend / Playwright tests

1. Run a second malicious-origin web server and prove it cannot log in, read, mutate, or log out the application session.
2. Expire the backend session and assert the next API 401 clears local auth state and redirects to login.
3. Modify local storage manually and prove protected data remains unavailable.
4. Render common stored-XSS payloads in every person field and prove they remain inert text.
5. Assert logout clears both server and browser state.
6. Inspect the production bundle and fail when it contains inline source maps, `eval(`, or known secret names.

### CI policy tests

- Backend: `./mvnw clean verify` with corrected aggregate coverage and a minimum threshold.
- Frontend: `npm ci`, lint, type/build, Playwright auth/security, and smoke suites.
- Supply chain: Maven dependency policy, `npm audit`, Trivy filesystem and image scans, Semgrep, secret scan, SBOM generation.
- Deployment: Docker build, non-root assertion, health-check assertion, image vulnerability scan, and policy checks for action SHA pinning and forbidden build-secret arguments.

## Review Limitations

- No production environment, reverse proxy, cloud account, GitHub administration settings, FOSSA dashboard, or Sonar dashboard was accessed.
- The final Docker image was not built or scanned because Docker was unavailable locally; source and Dockerfile scans were completed.
- This is a repository and local-runtime assessment, not a full penetration test. Network infrastructure, IAM policies, TLS configuration, backups, monitoring response, and production data controls remain outside the verified scope.
